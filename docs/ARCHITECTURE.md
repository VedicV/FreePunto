# Архітектура

## Напрям

Перша версія - нативний macOS-застосунок у рядку меню на Swift, SwiftUI та AppKit.

Головне завдання архітектури - надійні ручні перетворення тексту. Автокорекції, словників і фонового аналізу мови немає.

## Основні модулі

```text
Застосунок у рядку меню
  -> Контролер гарячих клавіш
  -> Визначення цілі тексту
  -> Двигун перетворень
      -> Перетворення розкладки
      -> Перетворення регістру
      -> Транслітерація
      -> Вибір наступної мови
  -> Заміна тексту
  -> Перемикання системної розкладки
  -> Сховище налаштувань
  -> Статусна іконка
  -> Діагностика дозволів
```

## Потік дії

```text
Користувач натискає гарячу клавішу
  -> контролер визначає команду
  -> застосунок бере виділений текст або останнє слово
  -> двигун будує заміну
  -> застосунок замінює текст
  -> застосунок перемикає системну розкладку на мову результату
  -> статусна іконка оновлює стан
```

Фонового потоку автокорекції немає.

## macOS API, які потрібно перевірити

### Гарячі клавіші

Потрібно перевірити:

- глобальні комбінації клавіш;
- одиночне натискання `Control`;
- відмінність одиночного натискання від комбінації з іншими клавішами;
- лівий і правий `Control`;
- конфлікт із системними комбінаціями клавіш.

Ризик: одиночний `Control` може конфліктувати із системними або користувацькими комбінаціями клавіш. Тому вибір гарячої клавіші має бути налаштуванням із самого початку.

### Читання виділеного тексту

Бажаний порядок:

1. Accessibility API для вибраного тексту, де це можливо.
2. Запасний спосіб через буфер обміну: скопіювати виділення, прочитати буфер обміну, відновити попередній вміст.

Запасний спосіб через буфер обміну має відновлювати попередній вміст і не зберігати його.

### Останнє слово

Можливі стратегії:

1. Короткий in-memory буфер нещодавно введених символів.
2. Виділення попереднього слова системною навігацією, читання, заміна і відновлення курсора.
3. Accessibility text range APIs, якщо конкретний застосунок їх підтримує.

Першу версію краще почати з виділеного тексту, а потім додати останнє слово після перевірки надійної заміни.

### Заміна тексту

Бажаний порядок:

1. Accessibility API replacement, де це можливо.
2. Заміна через вставлення з буфера обміну з відновленням попереднього вмісту.
3. Синтетичні key events тільки якщо іншого надійного шляху немає.

Заміна має за можливості потрапляти в нативний стек відкату цільового застосунку. Окремий власний відкат не реалізуємо.

### Перемикання системної розкладки

Після успішного перетворення потрібно перемкнути активну розкладку macOS на мову результату:

- результат англійський -> англійська розкладка;
- результат російський -> російська розкладка;
- результат український -> українська розкладка.

Для цього потрібно перевірити Text Input Source APIs:

- отримання доступних джерел введення;
- вибір джерела введення за мовою;
- поведінку, якщо потрібна розкладка не додана в налаштуваннях macOS;
- запасний сценарій: показати користувачу, що потрібну розкладку треба додати.

## Двигун перетворень

### Перетворення розкладки

Перетворює текст за фізичною клавіатурною картою:

- EN <-> RU;
- EN <-> UA;
- RU <-> UA через спільну карту фізичних клавіш.

Приклад:

```text
ghbdtn -> привет
руддщ -> hello
```

Це не транслітерація.

### Вибір наступної мови

Модуль визначає мову результату.

Послідовний режим:

```text
Англійська -> російська -> українська -> англійська
Російська  -> англійська -> українська -> російська
Українська -> англійська -> російська -> українська
```

Режим фіксованої цілі:

```text
Російська/українська -> англійська
Англійська -> вибрана кирилична мова
```

Для повторних натискань на одному й тому самому фрагменті бажано зберігати короткий контекст останнього перетворення, щоб цикл ішов передбачувано і міг повернути початковий текст.

### Перетворення регістру

Підтримуємо:

- нижній регістр;
- регістр речення;
- великі перші літери слів;
- виправлення тексту, випадково набраного з Caps Lock, у звичайний нижній регістр.

Окремий режим "усі літери великі" не потрібен.

### Транслітерація

Фонетична транслітерація:

- російська -> латиниця;
- українська -> латиниця;
- латиниця -> російська;
- латиниця -> українська.

Транслітерація і перетворення розкладки - різні операції та різні таблиці.

## Налаштування

```text
версія_налаштувань
увімкнено
запускати_при_вході
основна_гаряча_клавіша
гаряча_клавіша_регістру
гаряча_клавіша_транслітерації
гаряча_клавіша_паузи
режим_перемикання: послідовний | фіксована_ціль
цільова_мова_фіксованого_режиму: російська | українська
цільова_мова_транслітерації: російська | українська
режим_статусної_іконки
```

## Мінімальний інтерфейс

Використовуємо елемент у рядку меню і компактне вікно налаштувань.

Вікно налаштувань має містити тільки:

- гарячі клавіші;
- режим перемикання;
- цільову мову;
- цільову мову транслітерації;
- запуск при вході;
- статус дозволів macOS.

## Межа приватності

Оскільки автокорекції немає, застосунку не потрібно постійно аналізувати все введення.

Для останнього слова, якщо знадобиться буфер:

- зберігати тільки нещодавно введені символи;
- зберігати тільки в пам'яті;
- очищати при зміні фокуса;
- очищати при паузі;
- очищати при переході в сон і блокуванні екрана.

Вміст тексту не пишеться на диск і не надсилається в мережу.

## Тести

### Модульні тести

- Карти EN/RU/UA.
- Зворотність перетворень.
- Послідовний режим.
- Режим фіксованої цілі.
- Перемикання цільової розкладки.
- Нижній регістр.
- Регістр речення.
- Великі перші літери слів.
- Фонетична транслітерація.
- Збереження пунктуації та пробілів.

### Ручні інтеграційні тести

- TextEdit.
- Notes.
- Safari і Chrome.
- Адресний рядок браузера.
- Messages або Telegram.
- VS Code або інший редактор.
- Поля паролів: перевірити, що застосунок не надсилає нічого в мережу і поводиться настільки добре, наскільки дозволяють macOS API.
