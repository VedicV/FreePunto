# Безопасность и приватность

## Позиция по безопасности

Приложение не делает автокоррекцию и не должно вести себя как кейлоггер. Оно берет текст только для выполнения явной команды пользователя: преобразовать выделение или последнее слово.

Главное правило: текст не уходит в сеть и не сохраняется как история.

## Правила

- Нет телеметрии.
- Нет облачной обработки.
- Нет отправки текста в сеть.
- Нет истории нажатий клавиш.
- Нет истории введенного текста.
- Нет истории буфера обмена.
- Нет автокоррекции.
- Нет словарей и обучения на пользовательском тексте в первой версии.
- Нет сохранения исходного или преобразованного текста на диск.

## Работа с текстом

Обычный сценарий:

1. Пользователь нажимает горячую клавишу.
2. Приложение читает выделенный текст или последнее слово.
3. Приложение преобразует текст локально.
4. Приложение заменяет текст.
5. Приложение забывает текст.

Если для последнего слова понадобится короткий буфер, он должен быть:

- только в памяти;
- ограничен по длине;
- очищаться при смене фокуса;
- очищаться при паузе;
- очищаться при переходе в сон и блокировке экрана.

## Запасной способ через буфер обмена

Если буфер обмена используется для чтения или замены текста:

- сохранить прежнее содержимое буфера обмена;
- восстановить прежнее содержимое после операции;
- не логировать буфер обмена;
- не сохранять буфер обмена;
- не отправлять буфер обмена в сеть;
- аккуратно обработать ошибку, если восстановление буфера обмена не удалось.

## Поля паролей и защищенный ввод

В первой версии не делаем пользовательские исключения приложений и не вводим специальное правило "не работать в полях паролей".

При этом приложение не должно пытаться обходить системные ограничения macOS. Если macOS или конкретное приложение не дает прочитать или заменить текст в защищенном поле, команда должна просто не выполниться.

Ключевое требование: даже если пользователь преобразует текст в поле пароля, исходный и преобразованный текст остаются локально, не пишутся в историю и не отправляются в сеть.

## Разрешения macOS

Вероятно понадобятся:

- Accessibility - для чтения/замены выделенного текста и управления текстовыми полями;
- Input Monitoring - если одиночный `Control` или последнее слово потребуют отслеживания ввода.

Приложение должно объяснять, зачем нужно каждое разрешение. Если разрешения нет, приложение должно работать в урезанном режиме или показывать понятную причину, почему команда недоступна.

## Диагностика

Диагностика не должна содержать пользовательский текст.

Можно хранить:

- состояние разрешений;
- тип команды;
- код успеха или ошибки;
- время выполнения;
- текущий режим;
- язык результата.

Нельзя хранить:

- исходный текст;
- преобразованный текст;
- содержимое буфера обмена;
- историю ввода;
- полные заголовки активных окон.

## Сеть

Для основной работы приложению сеть не нужна.

В первой версии лучше вообще не добавлять сетевые функции. Обновления можно делать вручную через подписанные релизы.

## Релизная безопасность

- Подписать приложение.
- Нотаризовать приложение.
- Зафиксировать зависимости.
- Не добавлять автообновление, пока нет подписанного и проверяемого механизма обновлений.
